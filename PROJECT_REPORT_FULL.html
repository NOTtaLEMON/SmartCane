The Smart Cane Clip-On is a modular electronic augmentation for a standard white cane. It attaches sensor electronics to an existing cane and connects wirelessly to a smartphone, providing real-time obstacle detection, drop/step warnings, fall-triggered SOS alerting, and on-device computer vision — all without requiring the user to purchase specialised hardware beyond a modern Android phone.

Three Loosely Coupled Modules

Module A — Sensor Cane (ESP32)
Firmware

TF-Luna LiDAR (forward) + VL53L0X ToF (downward) + MPU6050 IMU. Streams dist_fwd,dist_drop,fall_flag at 10 Hz over USB serial or WiFi WebSocket. Powered by a 3.7 V Li-ion cell through an MT3608 boost converter.

Module B — Android Gateway
Kotlin / Android

Receives ESP32 packets, raises SMS SOS on fall, and runs on-device YOLOv8 TFLite pipeline (CameraX) for obstacle detection with TTS callouts.

Module C — Python Vision / Dashboard
Python / ML

Streamlit dashboard for live monitoring; Universal_Vision.py for IP-webcam YOLOv8; training pipelines producing the TFLite models deployed to Android.

2. System Architecture

The Smart Cane Clip-On is built as three loosely-coupled modules that communicate through small, well-defined message contracts rather than direct function calls. This separation is deliberate — it means the firmware can be debugged on a benchtop with USB serial, the Android app can be developed against the Mock data source without any hardware present, and the Python training pipeline produces models that drop into the Android assets/ folder as opaque files. No module needs to know the internal state of any other.

The diagram below captures this at a glance: each coloured cluster is a self-contained codebase with its own build system, deploy mechanism, and test loop. The arrows crossing module boundaries represent only the on-the-wire data formats (a 3-field CSV packet, a TFLite flatbuffer, and a WebSocket frame) — there is no shared library or RPC boundary between them.

3. Hardware Subsystem

The cane's electronics centre on a single ESP32 DevKit that reads three sensors, drives a buzzer + LCD for local feedback, and broadcasts a packet over either USB serial or WiFi WebSocket. The whole board is powered by a 3.7 V Li-ion cell stepped up to 5 V by an MT3608 boost converter, switched on/off by a panel-mounted KD slide switch, and housed in a 150 × 150 × 70 mm ABS enclosure that mounts to a standard white cane. Total parts cost sits well under ₹2,000.

The three sensors cover three distinct physical signals so that none of them duplicate information. TF-Luna LiDAR points forward and reports obstacle distance up to several metres. VL53L0X ToF points downward and reports ground distance, so a sudden increase means a step, kerb or pit. MPU6050 IMU tracks the cane's own acceleration, so a sharp spike means the cane has been dropped or the user has fallen. The earlier prototype also carried an LDR for ambient-light sensing, but it was removed once the design settled — the smartphone's own light sensor (and the user's TTS feedback) made the on-cane LDR redundant.

Firmware Architecture

Two firmware variants are maintained side-by-side:

Cane_Firmware.ino
Serial Baseline

USB Serial only. Outputs dist_fwd,dist_drop,fall_flag at 10 Hz on Serial @ 115200. Zero WiFi overhead — maximum determinism for benchtop testing.

Cane_Firmware_WiFi.ino
WiFi v2.1

Adds WiFi.h + WebSocketsServer on port 81. Broadcasts the same packet to all connected clients. Non-blocking WiFi reconnect via lastWifiRetry gate.

Key Firmware Design Decisions

⏱️

Non-blocking Main Loop

Single LOOP_MS = 50 cadence drives all sensors. WiFi reconnect, buzzer toggling, and LCD IP refresh are time-gated via millis() checkpoints (lastTick, lastWifiRetry, lastBuzzerOff, lastIPDisplay). A single slow sensor cannot stall the others.

🔔

Priority-Based Buzzer Logic

Three alert sources arbitrated in strict priority order: Fall (continuous 5 s tone) overrides LiDAR obstacle (fast / medium / slow beeps at the 500 / 1000 / 1500 mm bands) which overrides the ToF drop warning (path-clear but ground gone). Higher-priority events always preempt lower ones because a missed fall is a much worse outcome than a missed distance band.

🔌

Library-free MPU6050

Direct Wire register access (PWR_MGMT_1 = 0x6B, ACCEL_XOUT_H = 0x3B) avoids dependency conflicts with Adafruit/Electronic Cats MPU libraries that clashed with the ESP32 core version.

📡

Manual TF-Luna 9-byte Frame Parser

The 0x59 0x59 … frame is parsed inline rather than via a vendor library, making the build portable across all ESP32 board package versions and eliminating an unreliable third-party dependency.

How the priority works: every 50 ms the firmware first checks the IMU — a fall always wins and locks the buzzer to a continuous 5-second alarm. If there is no fall, the forward LiDAR distance picks one of three beep speeds (fast / medium / slow). Only when both checks say “all clear” does the downward ToF get a chance to flag a drop ahead. The full sensor packet is broadcast on every tick regardless of which branch wins, so the phone dashboard always sees the raw data.

5. Software Subsystem — Android App

The Android app serves a dual role: it is both the gateway that listens to the cane and dispatches SMS-based emergency alerts, and the vision engine that runs YOLOv8 inference on the phone's rear camera. Splitting these two responsibilities across an Activity and a Foreground Service is a deliberate Android-architecture choice. The vision Activity is bound to the user's screen — it can be paused or backgrounded freely, and CameraX will release the camera cleanly. The SOS Service, on the other hand, must keep running with the screen off because falls do not wait for the user to unlock their phone. Both communicate with the live dashboard through LocalBroadcastManager, which is in-process, fast, and does not leak across app boundaries.

The TFLite model loader (TfliteObjectDetector.kt) is the most defensive piece of the codebase because it has to deal with three independent sources of variation: (a) models may be detection-only (84-channel output) or segmentation (116-channel output); (b) models may target the COCO 80-class taxonomy or our custom 8-class taxonomy; and (c) the device may or may not support the GPU delegate. Rather than ship four separate code paths, the loader inspects the TFLite tensor shapes at init time and branches into the correct post-processor automatically, with the GPU delegate wrapped in runCatching so a missing OpenGL ES 3.1 stack falls back silently to 4-thread CPU inference.

Android Methodology Highlights

🔀

Foreground Service + LocalBroadcastManager Decoupling

CaneSosService handles all networking and SOS logic independently of the UI. PhoneDashboardActivity subscribes via LocalBroadcastManager, so the SOS pipeline keeps running with the screen off or activity in background.

🛡️

Defensive Model Loader with Candidate List

TfliteObjectDetector.loadModel() iterates MODEL_CANDIDATES: smartcane_segmentation_320_float16 → yolov8n_seg_320_float16 → yolov8n_320_float16 → …, skipping any file under 1 KB (stub placeholder). APK remains robust to missing or partial assets.

⚡

GPU Delegate with CPU Fallback

Each detector tries GpuDelegate() inside runCatching and falls back to 4 CPU threads on unsupported devices or OpenGL ES < 3.1. No crash on entry-level devices.

🎙️

TTS Rate-Limiting per Label

ALERT_COOLDOWN_MS = 3000 per label prevents speech spamming when the same obstacle stays in frame. Each label has its own cooldown clock so multiple simultaneous obstacles each speak at their own 3 s cadence.

📱

API-Level-Aware Permission Matrix

Conditional buildList per API level in MainActivity: BLE runtime permissions added at API 31+, POST_NOTIFICATIONS added at API 33+. Avoids asking for permissions that don't exist on older devices.

TTS Alert Thresholds & Cooldown Policy

The single unified 8-class model was deprecated mid-project in favour of seven specialized YOLOv8n detectors running in parallel, one per hazard class. This change was driven by an empirical observation: individually fine-tuned per-class models consistently outperformed a multi-task unified head on the small, imbalanced datasets we had access to. The parallel-detector design also means a poor-performing class can be retrained or swapped without touching the others.

Voice alerts are emitted only for the highest-confidence detection in each frame that clears its class threshold and its per-label cooldown — this matches what is shown first on screen and prevents audio flooding when many objects are visible. Push notifications were tried in an earlier iteration and removed: in noisy walking scenarios they buried the user under banner spam without adding information beyond what TTS already conveyed.

7. ML Model Training Pipeline

Computer vision is the highest-leverage subsystem in the project — it is what turns a basic distance-sensing cane into something that can describe the world around the user. The challenge is that on a phone, every millisecond of inference latency is paid for in battery, in user response time, and in the smoothness of the audio feedback loop. The pipeline below is engineered around the constraint that any frame the user is going to hear about must be inferenced inside the camera's 33 ms frame budget, which is why we settled on 320×320 input resolution and float16 weights.

A second hard constraint is that real annotated data for visually-impaired-relevant obstacles (potholes from a pedestrian's viewpoint, kerbs at cane-tip height, e-scooters on pavements) is scarce and slow to collect. We work around this with a validate-pipeline-then-collect-data workflow: synthetic data is used to prove the entire chain — Roboflow export → YOLO training → TFLite conversion → Android assets → on-device inference — works end-to-end before a single real photograph is annotated. That way, when production data does arrive, deploying it is a one-command exercise rather than an integration project.

Strategy: Validate-pipeline-then-collect-data. Synthetic data (generate_sample_dataset.py) is used to prove that training → TFLite export → Android deployment works end-to-end before investing weeks in real annotation. Models are exported at 320×320 float16 to fit the mobile inference budget (~200–300 ms/frame) while keeping the APK under a shippable size.

8. Python Dashboard & Vision

The Python side of the project plays two distinct roles. Project_Dashboard.py is a Streamlit application that gives the development team a live, visual view of what the cane is reporting — distance bands, fall flags, ambient-light levels — and is invaluable during demos and threshold tuning. Universal_Vision.py is a desktop YOLOv8 runner that consumes an IP-Webcam stream from any modern smartphone, letting us validate model quality on full-resolution video before committing to a TFLite conversion. Both share the same packet parser, which means a change to the wire format is a one-file edit.

The dashboard's three connection modes — Serial, WiFi, and Mock — exist so that any developer can work on the visualization layer without needing the physical cane. Mock mode in particular has been invaluable for UI work: it synthesises realistic packets including fall events and critical-distance crossings, so chart styling, alert banners, and zone-band rendering can all be exercised against deterministic data.

10. Challenges Faced & Resolutions

Hardware / Firmware Challenges

🔌

I2C Bus Contention Between Three Devices

VL53L0X (0x29), MPU6050 (0x68), and LCD (0x27) all share SDA/SCL on GPIO 21/22. With long wires and pull-up resistors at both device ends, the bus would occasionally lock up.

Verified non-overlapping addresses. Added single pull-up pair at ESP32 side only. Confirmed wire lengths under 30 cm.
📡

TF-Luna Intermittent Frame Drops Under Load

When the MCU was busy with WiFi operations or I2C reads, the UART buffer would overflow and the 9-byte 0x59 0x59 frame would partially arrive, producing corrupted distances.

Moved TF-Luna read to inline while-loop inside loop() with a 9-byte minimum guard. Increased UART2 RX buffer size. Non-blocking main loop prevents starvation.
🛠️

MPU6050 Arduino Library Conflicts

Both the Adafruit MPU6050 and Electronic Cats MPU6050 libraries conflicted with the ESP32 Arduino core version being used, causing compile errors on multiple machines.

Replaced all library calls with direct Wire register reads: wake MPU via PWR_MGMT_1=0x00, read 6 bytes from ACCEL_XOUT_H. Eliminated the dependency entirely.
📶

WiFi Connection Blocking the Sensor Loop

The initial firmware had WiFi.begin() in a blocking while loop waiting for connection. During a WiFi outage, the sensor loop would stall for up to 10 s, causing missed buzzer beeps.

WiFi connect attempt is time-boxed in setup() to 10 s with millis() gate. Runtime reconnects are non-blocking via lastWifiRetry flag checked each loop iteration.
📻

ESP32 5 GHz Band Incompatibility

ESP32 only supports 2.4 GHz WiFi. Several lab access points used dual-band SSIDs that would advertise both bands. Devices connecting to the 5 GHz band would never see the ESP32.

Documented in ESP32_WIFI_CONNECTION_GUIDE.md — users must configure phone hotspot to 2.4 GHz only or select a 2.4 GHz-only SSID.
🤸

False Fall Triggers from Sharp Movement

Raising the cane quickly over a step or waving the arm produced accel magnitude spikes above 20000 that triggered false fall alerts and SMS messages.

Threshold tuned empirically via FallTriggerTest.kt. Added fall-flag re-arm condition: flag only re-set when accel returns to baseline <10000, preventing rapid re-trigger.

Android / Software Challenges

🧩

TFLite Tensor Shape Varied Between Models

Detection output [1,84,2100] (4 bbox + 80 COCO classes) vs segmentation [1,116,2100] (4 bbox + 80 classes + 32 mask coeffs) vs custom 8-class [1,12,2100] — each required different post-processing.

TfliteObjectDetector inspects output shape at init time and branches into detect/seg/custom post-processing. Single loader handles all variants transparently.
🗑️

Stub .tflite Assets (29-byte Placeholders) Crashing Inference

Early in development, placeholder empty files were committed to git for the asset paths before real models were trained. The model loader would open these files and crash on the first inference call.

MODEL_CANDIDATES ordering + declaredLength < 1024 bytes skip guard. Loader silently skips any asset under 1 KB and tries the next candidate.
💻

GPU Delegate Failures on Older Android Devices

Devices without OpenGL ES 3.1 or dedicated GPU compute units would throw on GpuDelegate() construction, crashing the vision activity entirely.

Wrapped GpuDelegate() in runCatching with 4-thread CPU fallback. Zero crash path — worst case is slower inference.
🔊

TTS Spamming the Same Label Every Frame

At 30 fps with a person in frame, the TTS engine would queue "Person ahead" 30 times per second, causing incomprehensible audio output.

Per-label ALERT_COOLDOWN_MS = 3000 map. Each label tracks its last-spoken timestamp independently.
📨

SMS Spamming on Continuous Fall Flag

If the user fell and the cane continued reading high accel values, the SOS SMS could be sent hundreds of times to emergency contacts.

60 s SMS_COOLDOWN_MS rate-limit inside CaneSosService. Single atomic lastSmsSent timestamp checked before every send.
🔄

Streamlit Reruns Killing WebSocket Connections

Streamlit re-executes the entire script on every user interaction. Each rerun would create a new WebSocket connection and close the old one, causing disconnect/reconnect cycles every second.

Connection caching keyed by (mode, ip) in Streamlit session_state. Reruns detect the existing live socket and reuse it.
📁

Roboflow data.yaml Relative Paths Breaking Training

Roboflow exports data.yaml with ../train/images relative paths. When training from a different working directory (e.g. project root vs data/pothole/), YOLO would fail to find the images.

fix_data_yaml() function in prepare scripts rewrites all paths to absolute and emits data_abs.yaml. Training always uses the absolute-path version.
