1 Executive Summary

The Smart Cane Clip-On is a modular assistive device that augments a traditional white cane with electronic obstacle perception, drop detection, fall alerting, and on-device computer vision. The system is composed of three loosely-coupled modules:

Module A
Sensor Cane (ESP32)
  • TF-Luna LiDAR — forward range
  • VL53L0X ToF — drop detection
  • MPU6050 — fall detection
  • LDR — ambient light
  • USB Serial + WiFi WebSocket output
Module B
Android Gateway App
  • Foreground service → ESP32 WebSocket
  • On-device YOLOv8 TFLite inference
  • CameraX + TTS obstacle callouts
  • GPS + SMS SOS on fall detection
  • Phone dashboard UI
Module C
Python Vision / Dashboard
  • Streamlit dashboard (Serial / WiFi / Mock)
  • Universal_Vision.py — YOLOv8 on IP Webcam/RTSP
  • Training pipelines for TFLite models
  • Dataset prep + synthetic data generator

Data protocol shared across all transports: dist_fwd,dist_drop,fall_flag,light_val — e.g. 045,180,0,0550 at 10 Hz.

3.2 Firmware Methodology

Two firmware variants are maintained side-by-side:

  • Cane_Firmware.ino — USB serial baseline; emits L:<cm>,T:<mm>,LDR:<val>,F:<0|1> at 10 Hz on Serial @ 115200.
  • Cane_Firmware_WiFi.ino (v2.1 NON-BLOCKING) — adds WiFi.h + WebSocketsServer on port 81, broadcasts the same packet to all connected clients, prints the assigned IP address to LCD and Serial.

Key design decisions:

  1. Non-blocking main loop. A single LOOP_MS = 50 cadence drives all sensors; WiFi reconnect, buzzer toggling, and LCD IP refresh are gated via millis() timestamps so no single operation can stall the loop.
  2. Tiered buzzer logic. Distance bands map to beep intervals of 150 / 400 / 800 ms. Fall detection overrides to a continuous tone — safety-critical signals always win.
  3. Library-free MPU6050. Direct Wire register access (PWR_MGMT_1 = 0x6B, ACCEL_XOUT_H = 0x3B) avoids dependency conflicts with competing IMU libraries on the ESP32 core.
  4. Manual TF-Luna framing. The 9-byte 0x59 0x59 … frame is parsed inline, keeping the build portable across ESP32 board package versions.
  5. Configurable thresholds at the top of each file: LDR_DARK_THRESHOLD = 700, FALL_THRESHOLD = 20000 (raw accel sum).

3.3 Hardware Challenges & Resolutions

I2C bus contention between VL53L0X (0x29), MPU6050 (0x68), and LCD (0x27)
Verified non-overlapping I2C addresses; shared SDA/SCL on GPIO 21/22 with pull-ups for long wire runs.
TF-Luna intermittent frame drops when MCU was busy
Switched to inline 9-byte framing inside loop() with a while (available() >= 9) drain pattern.
MPU6050 library conflicts with the ESP32 Arduino core
Replaced all library calls with direct Wire register reads — zero external dependency.
WiFi connect blocking the sensor loop on startup
Connect attempt is time-boxed in setup(); runtime reconnects are non-blocking via lastWifiRetry millis gate.
ESP32 does not support 5 GHz WiFi
Explicitly documented in the connection guide; only 2.4 GHz SSIDs accepted.
False fall triggers from sharp arm movement
Raw accel-sum threshold (20 000) tuned empirically; fall flag re-arms only when acceleration returns to baseline.
CH340/CP2102 USB driver missing on Windows developer machines
Driver links and install steps captured in FIRMWARE_SETUP.md and the WiFi connection guide.

4.2 Methodology Highlights

  1. Foreground service + LocalBroadcastManager decouples networking (CaneSosService) from UI (PhoneDashboardActivity), so the SOS pipeline keeps running with the screen off.
  2. Defensive model loader. TfliteObjectDetector.loadModel() iterates a MODEL_CANDIDATES list and skips any stub file under 1 KB — robust to missing or partial assets.
  3. GPU delegate with CPU fallback. Each detector calls runCatching { GpuDelegate() } and falls back to 4 CPU threads on unsupported devices.
  4. Per-class confidence thresholds follow a safety-first principle: person/vehicle = 0.25–0.30 (aggressive), furniture/animal = 0.35 (medium), hazards = 0.50 (conservative), clear_path = 0.60 (very conservative — no false reassurance).
  5. TTS rate-limit. ALERT_COOLDOWN_MS = 3000 per label prevents speech spamming when the same obstacle stays in frame.
  6. SMS rate-limit. SMS_COOLDOWN_MS = 60 000 inside CaneSosService prevents SOS spam on sustained fall flags.
  7. Conditional permission list built at runtime via buildList per API level: BLE runtime perms added on API 31+, POST_NOTIFICATIONS on API 33+.

4.3 Android Software Challenges & Resolutions

TFLite tensor shape varies: detect [1,84,2100] vs seg [1,116,2100] vs custom 8-class
Single loader inspects output shape at init and branches into the correct post-processing path and label map.
Stub .tflite placeholder assets (29 bytes) crashing inference on first load
MODEL_CANDIDATES order + declaredLength < 1024 skip guard in the loader.
GPU delegate failures on devices without OpenGL ES 3.1
runCatching { GpuDelegate() } with silent fallback to 4-thread CPU interpreter.
Permission matrix churn across API 26–34
Conditional buildList per API level in both MainActivity and PhoneDashboardActivity.
APK ships multiple model assets → large APK size
Float16 quantization at 320 px keeps each model ≈6 MB; unused models are never mmapped into memory.

5.1 Dashboard Architecture

Three connection modes (Serial / WiFi / Mock) share a single Packet dataclass and parser. The same downstream charting and zone-classification code runs regardless of transport. WiFiSource caches its WebSocket connection across Streamlit reruns to avoid reconnect overhead. A mode-aware status banner (🎮 Mock / 📡 WiFi LIVE / 🔌 Serial LIVE) prevents misreading mock data as real telemetry.

8 Methodology Summary

Hardware — Sensor-Fusion-First Design

Each sensor owns exactly one failure mode: LiDAR for forward range, ToF for drop, IMU for fall, LDR for darkness. The firmware is intentionally library-light (raw I2C registers, manual TF-Luna framing) to maximise portability across ESP32 core versions, and the main loop is non-blocking so a slow sensor cannot starve the others. The serial protocol is a fixed-width CSV that doubles as the WebSocket payload, so the same parser serves every consumer — laptop dashboard, Android service, and mock generator alike.

Software — Three-Tier Separation

  1. Transport layer (SerialSource / WiFiSource / MockSource in Python; CaneSosService WebSocket client in Kotlin) — pluggable, swappable at runtime without touching business logic.
  2. Perception layer (TFLite models with auto-detected tensor shape and class count, per-class thresholds, NMS, GPU-with-CPU-fallback) — multiple specialised detectors run cooperatively (TfliteObjectDetector + PotholeDetector + ElectricPoleDetector) rather than one monolithic model.
  3. Action layer (TTS callouts, tiered buzzer, SMS SOS with cooldown, dashboard zone banners) — every output channel is rate-limited so the user is never overwhelmed and the SMS gateway is never abused.

ML — Validate Pipeline First, Collect Data Second

Synthetic data (generate_sample_dataset.py) is used to prove that training → TFLite export → Android deployment works end-to-end before investing weeks in real-world annotation. Models are exported at 320×320 float16 to fit the mobile inference budget (~200–300 ms/frame) while keeping the APK shippable at ≈30 MB total for four bundled models.
